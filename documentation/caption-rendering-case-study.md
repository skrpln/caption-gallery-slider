# Caption Rendering Case Study

Связанные заметки: [[documentation/phase-2-captions]], [[progress_log]], [[log/2026-05-27_00-48_phase-2-hardening-and-close]]

## Короткая история

В Phase 2 мы добавили к галерее редактируемые подписи: каждая подпись хранится как обычная Markdown-заметка, а внутри виджета её можно открыть, отредактировать и увидеть как rendered Markdown. На бумаге задача выглядела простой: прочитать body caption-файла, передать его в `MarkdownRenderer.render()` и показать результат в компактной панели под изображением.

На практике именно этот участок стал одним из самых показательных мест проекта. Нам нужно было совместить три разных режима, которые обычно живут отдельно:

- обычную Obsidian-заметку со своим Markdown preview;
- inline editing через `contenteditable`;
- компактный preview внутри кастомного DOM-виджета.

Главная сложность была не в том, чтобы Markdown отрендерился. Он отрендерился сразу. Сложность была в том, чтобы он выглядел нативно и при этом оставался компактным.

## Симптом

В режиме редактирования подпись выглядела правильно: строки шли плотно, переносы были предсказуемыми, текст помещался в маленькую caption panel. Но после выхода из редактирования тот же самый текст в режиме просмотра начинал занимать гораздо больше высоты. Между строками появлялись огромные интервалы, из-за которых фиксированная область подписи почти сразу превращалась в прокручиваемый фрагмент.

Сначала казалось, что проблема в `line-height` или в стандартных margin у Markdown-блоков. Мы уменьшали line-height, обнуляли `margin-block` у `p`, `ul`, `h1-h6`, добавляли правила для Obsidian wrapper-классов вроде `.el-p` и `.markdown-preview-section > div`. Это помогало неустойчиво или не помогало вообще.

## Ложная гипотеза

В какой-то момент появилась правдоподобная версия: возможно, при сохранении из `contenteditable` мы случайно задваиваем переносы строк. Если вместо одиночных `\n` в caption-файл попадают пустые строки, MarkdownRenderer честно превращает их в отдельные абзацы, и визуальный интервал становится большим.

Мы проверили реальный caption-файл построчно и побайтно. В файле не было лишних пустых строк между обычными строками текста. Например:

```md
тест [[ссылка не работает]] [[note for popup gallery test]]
тут можно нормально вставлятьс сылки [[progress_log]]
- [ ] длинная подпись
- [ ] с булетпойнтами

# и даже заголовком
как отобразится интересно
```

Одиночные переносы были одиночными. Значит, данные были чистыми, а проблема возникала уже на этапе отображения.

## Настоящая причина

Ключ оказался в CSS-свойстве `white-space`.

Для режима редактирования мы использовали:

```css
white-space: pre-wrap;
```

Это правильное поведение для plain text editor: пользователь нажимает Enter, перенос виден; несколько пробелов и переносы не теряются; `contenteditable` ощущается ближе к обычному текстовому полю.

Но тот же контейнер использовался и для rendered Markdown. После `MarkdownRenderer.render()` внутри caption panel появлялся не простой текст, а HTML-структура Obsidian: абзацы, ссылки, списки, заголовки, служебные wrapper-элементы и whitespace text nodes между DOM-узлами. В обычном Markdown preview эти служебные переносы HTML схлопываются браузером. А внутри контейнера с `white-space: pre-wrap` они могут начать вести себя как настоящие визуальные переносы.

Иными словами, мы пытались одним CSS-режимом обслужить два разных представления:

- editing mode — plain text, где `pre-wrap` полезен;
- preview mode — rendered HTML, где `pre-wrap` начинает мешать.

## Решение

Мы разделили режимы:

```css
.og-gallery__caption-content {
  white-space: normal;
}

.og-gallery__caption.is-editing .og-gallery__caption-content {
  white-space: pre-wrap;
}
```

В preview mode контейнер теперь ведёт себя как обычный Markdown preview: HTML whitespace схлопывается, а структуру текста определяет MarkdownRenderer. В editing mode сохраняется поведение текстового поля, где переносы пользователя видны буквально.

Дополнительно мы оставили компактные правила для Obsidian block wrappers после `MarkdownRenderer.render()`:

- `margin-block: 0`;
- `min-height: 0`;
- компактный `line-height`.

Это нужно не для исправления главной причины, а для того, чтобы caption panel оставалась плотной и не зависела слишком сильно от темы.

## Вторая тонкость

Когда мы слишком агрессивно уплотняли preview, мы временно сломали Markdown-заголовки: `# heading` почти не отличался от обычного текста. Причина была в нашем же post-processing: мы принудительно ставили `font-size: inherit` на `h1-h6`.

Это оказалось неправильным компромиссом. Компактность caption panel не должна уничтожать Markdown-иерархию. Итоговое решение:

- обнулять отступы и wrapper-высоты;
- не трогать `font-size` заголовков;
- оставить размеры заголовков теме Obsidian.

Так подпись остаётся компактной, но Markdown продолжает быть Markdown.

## Почему это удачное решение

Главный вывод: inline editing и rendered Markdown нельзя считать двумя состояниями одного и того же DOM-контейнера с одинаковой типографикой. Это два разных режима отображения, и им нужны разные правила whitespace.

Удачным решением стало не “закрутить CSS посильнее”, а разделить ответственность:

- MarkdownRenderer отвечает за Markdown semantics;
- caption CSS отвечает за границы компактной панели;
- editing mode получает literal whitespace;
- preview mode получает обычное HTML whitespace collapsing.

После этого поведение стало предсказуемым: подпись в виджете выглядит компактно, редактирование не прыгает визуально, ссылки и заголовки сохраняют привычное Obsidian-поведение.

## Урок для статьи

Если в Obsidian-плагине нужно отрендерить Markdown внутри собственного виджета, стоит отдельно проверить `white-space` на контейнере. Свойство, полезное для `contenteditable`, может неожиданно испортить rendered Markdown, потому что начинает визуализировать служебные переносы, которые в обычном preview не видны.

Этот баг был особенно коварным, потому что выглядел как проблема line-height или margin. Но настоящая причина была на уровень ниже: не размер строки, а режим обработки whitespace.
